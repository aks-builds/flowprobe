# FlowProbe UI v2 — Design Spec
**Date:** 2026-06-21
**Status:** Approved
**Approved visual:** Warm Hybrid (C) + Live Pipeline event stream (A) + Instrument Panel spark-lines + run counter (B)

---

## 1. Design Philosophy

**Structured instrument during calm. Alive during a run.**

- **Arctic Light base** — white surface, slate borders, violet accent. Clean, never cluttered.
- **Animation = semantic signal** — status dots animate only during active states; stopping the animation IS the completion signal (Vercel Geist pattern, confirmed high-confidence). No decorative animation.
- **Navigation chrome subordinated** — sidebar is dimmer than the work surface (Linear 2026 pattern). Icon-first, text on hover/expand.
- **Desktop-first interactions** — no `cursor: pointer` on list items (Raycast pattern). No web-style hover highlights on rows. Native feel.
- **Zero silent failures** — every invalid state shows an inline non-blocking message. No crash, no freeze, no empty screen.

---

## 2. Three Functional Flows

### Flow 1: Run Flow

**Trigger:** "▶ Run Collection" button in titlebar.

**Pre-run validation (blocking — shown inline, run disabled until resolved):**
- Collection must be loaded (not empty state)
- Active flow must have at least 1 step
- Every `producer` step must have `broker` and `topic` non-empty
- Every `http-assert` step must have a valid URL (starts with `http://` or `https://`)
- Every `wait` step must have `timeoutMs` > 0
- Broker used in flow must be connected (not disconnected status)
- Show validation errors as red badges on step cards + a dismissible summary banner above the canvas: `"2 steps have errors — fix them before running"`

**On Run:**
1. Run button changes to `■ Stop` (red) immediately on click
2. Titlebar run counter increments: `#47 → #48`
3. Run counter turns blue (running)
4. Canvas transitions to **Vercel-style timeline** (replaces the step pipeline view)
5. Each step row appears as: pulsing blue dot → green ✓ (on pass) / red ✕ (on fail)
6. Animation-as-liveness: dot pulses during active step; goes static on terminal state. No other motion.
7. A-style **event stream drawer** slides up from bottom (collapsible, 90px default height):
   - Shows raw message events in monospace: timestamp, direction arrow, broker/topic, offset/lag
   - Uses `Tauri Channel<T>` — NOT `emit()` (event system has 200ms latency; Channel is designed for ordered high-speed delivery per Tauri v2 docs)
   - Dark (#0f172a) background, muted text, live blinking cursor on last line
8. Step completion: timeline row fades in with `fadeScale` (150ms ease-out), duration appears in monospace right column
9. On partial failure: failed rows pin to top of timeline, a `"1 failed · 3 passed"` badge replaces the progress bar
10. On abort (Stop button): all pending rows show grey dash, banner: `"Run stopped — partial results"`
11. On all pass: animated gradient confetti bar appears under titlebar for 2s then fades; run counter turns green

**Runtime error handling:**
- Broker disconnects mid-run → current step fails with `"Broker connection lost"` in red, run halts, event stream shows the disconnect timestamp
- Step timeout → step fails with `"Timeout after Nms waiting for message"` in red
- HTTP assertion fails → step fails, red ✕ with assertion detail inline: `expected $.reserved === true, got false`
- DB connection unavailable → step fails with `"DB connection not registered"` + link to configure
- Any unhandled Rust-side panic → caught by Tauri error boundary, shown as banner: `"Run error: <message>"` — never crashes the window

### Flow 2: Collection Builder Flow

**Step inline editing (Linear-style — no modal):**
- Click a step card → card expands in place to show editable fields
- Fields: all step-type-specific properties (topic, broker, URL, timeout, assertions, query, etc.)
- Tab between fields, Enter to save, Esc to cancel
- `✎ Editing: <step-id>` label appears at top of the expanded card in accent color
- Keyboard hint shown: `↵ save · Esc cancel · Tab next field`
- On save: card collapses with a brief `fadeScale` confirmation (150ms)
- On Esc: card collapses with no animation (instant, cancellation feels fast)

**Per-field validation (inline, not blocking):**
- Empty required field → red border on that input + `"Required"` label below
- Invalid URL → `"Must start with http:// or https://"`
- `timeoutMs` = 0 or negative → `"Must be > 0"`
- Duplicate `step.id` → `"Step ID already used in this flow"`
- `broker` reference not in broker list → `"Broker not found — add it in the sidebar"`
- Assertion `path` is empty when type is `jsonpath` → `"JSONPath expression required"`
- Save button disabled while any field has an error

**Step type picker (Raycast-style mini palette):**
- Click `+ Add step` → a 200px wide popover appears inline (not modal)
- Lists all 5 step types with icon, name, one-line description
- Keyboard-navigable (↑↓, Enter)
- Filter by typing
- Escape closes without adding
- Adding a step: card appears with `fadeScale` + `shake` if it spawns with validation errors (empty required fields)

**Drag-and-drop reorder:**
- Dragging shows a semi-transparent ghost (opacity 0.5)
- Other cards shift to show a 4px violet drop-zone gap between them
- Drop: target card animates to new position with spring (200ms ease-out)
- No-drop zone (outside the flow row): cursor changes to `not-allowed`, ghost snaps back

**Collection management validation:**
- Cannot save a collection with 0 flows
- Cannot save a flow with 0 steps
- Collection name cannot be empty
- Empty collection shows: `"No flows yet — add one to get started"` with a centered CTA button

### Flow 3: Broker Connection Flow

**Broker status dots (Vercel Geist pattern):**
- Grey (static) = offline / not configured
- Amber (pulsing 1s ease-in-out infinite) = connecting / testing
- Blue (pulsing faster, 0.6s) = connected, sending data
- Green (static) = connected, idle
- Red (static) = error / connection refused

**Live latency display:**
- Connected brokers show latency next to the dot in monospace: `12ms`
- Latency updates every 5s via a background ping
- Latency > 500ms: amber text. Latency > 2000ms: red text with tooltip `"High latency — check broker health"`

**Broker configuration inline panel (no modal):**
- Click a broker row → sidebar expands a mini config form below the row
- Fields: URL/brokers, auth type, credentials (shown as `••••••`, toggle to reveal)
- Connect/Disconnect buttons
- `Test Connection` button → dot animates to amber → green (success) or red (failure) with error message: `"Connection refused at kafka-local:9092"` or `"SASL auth failed: invalid credentials"`
- All credential fields: empty → `"Required"` inline error. Invalid format → format hint.

**Broker errors during run surfaced in event stream:**
- `"14:31:02.881 ✕ kafka-local disconnected during step 'await-consume'"` in red
- The run timeline step row shows `"Broker connection lost"` as the failure reason

## 3. Command Palette Improvements

**Keyboard navigation:** ↑↓ to highlight, Enter to select, Escape to close.

**Inline keyboard shortcuts (Linear pattern):**
Every action shows its shortcut right-aligned: `Run Collection` → `Ctrl+R`, `New Flow` → `Ctrl+N`, `Validate` → `Ctrl+Shift+V`

**Expanded search:**
- Searches: collection names, flow names, step IDs, step types, broker names
- Results grouped by section: `Collections`, `Flows`, `Steps`, `Brokers`
- Section headers in small-caps muted text

**Empty state:**
- No results: `"No results for '${query}'"` with a suggestion: `"Try searching for a step type like 'kafka' or 'webhook'"`
- No collections loaded: `"No collections open — drag a .flowprobe.json file here or use File → Open"`

## 4. Validation & Error System

**Principles:**
- Never crash the window. Every Rust panic caught and surfaced as a banner.
- Never freeze the UI. All I/O via Rust async + `Tauri Channel<T>` for streaming.
- Inline errors, not blocking modals. Errors appear next to the field/step that caused them.
- Dismissible global banners for run-level errors (red, auto-dismiss after 8s or manual X).
- Warning toasts for non-blocking issues (amber, auto-dismiss 4s).

**Error component hierarchy:**
1. **Field error** — red border + small red text below the field. Shown on blur or save attempt.
2. **Step error badge** — red dot on step card corner. Shown when step has validation errors.
3. **Flow error banner** — `"N steps have errors"` above the canvas. Dismissible.
4. **Run error banner** — `"Run error: <message>"` at top of timeline. Persists until dismissed or next run.
5. **Global crash banner** — `"An unexpected error occurred: <message>"` at very top, red, with `"Copy details"` button.

**Specific error messages (all must be exact and actionable):**
- Empty collection on run: `"No collection loaded. Open a .flowprobe.json file to get started."`
- Flow has 0 steps: `"This flow has no steps. Add at least one step before running."`
- Step missing broker: `"'publish-event' is missing a broker. Select a connected broker."`
- Broker not connected: `"kafka-local is not connected. Click it in the sidebar to connect."`
- HTTP URL invalid: `"URL must start with http:// or https://"`
- Timeout expired: `"Step 'await-consume' timed out after 5000ms. No message received on topic order-events."`
- Assertion failed: `"Step 'check-inventory': $.reserved expected true, got false (GET /inventory/ord_8821 → 200)"`
- DB not configured: `"No database connection registered for '{{DB_CONN_ID}}'. Add it in Settings → Databases."`
- Kafka connection refused: `"Cannot connect to kafka-local:9092. Connection refused — is the broker running?"`
- SASL auth failure: `"Authentication failed for kafka-local. Check SASL username and password."`
- Rust panic caught: `"Internal error: <panic_message>. The UI is still running. Please report this issue."`

## 5. Technical Decisions

### Tauri Channel<T> for streaming (CRITICAL)
The Tauri event system (`emit()`) is **not suitable** for live log streaming — it has documented 200ms+ latency and JSON-string-only payloads (Tauri v2 official docs, confirmed 3-0 adversarial vote). All run progress must use `Channel<T>`:
```rust
// Rust side
#[tauri::command]
async fn run_collection(channel: Channel<RunEvent>, ...) {
    channel.send(RunEvent::StepStart { id: "publish-event".into() }).unwrap();
    // ... perform step ...
    channel.send(RunEvent::StepDone { id, passed, duration_ms, detail }).unwrap();
}
```
```typescript
// Svelte side
await invoke('run_collection', {
  channel: new Channel<RunEvent>(event => {
    // update timeline reactively, no latency
  })
})
```

### svelte-treeview for collection tree
Use `svelte-treeview` v4.8+ for the sidebar collection/flow tree. It implements three rendering modes:
- `recursive` for < 500 nodes (default FlowProbe case)
- `progressive` with exponential batching (20→40→80→160→320→500 per frame) for 500–10,000 nodes
- `virtual scroll` for 10,000+
This prevents the sidebar from freezing on collections with many flows.

### Svelte error boundaries
Wrap each major component (Sidebar, FlowCanvas, ResultPanel, CommandPalette) in a Svelte error boundary. On component error: show `"Something went wrong in <ComponentName>. Click to reload."` — never a white screen.

### Animation durations (from design tokens)
- Status dot: `--dur-fast` (150ms) for pass/fail transition
- Step card entrance: `--dur-fast` (150ms) `ease-out` (fadeScale)
- Timeline row entrance: `--dur-fast` staggered 30ms per row
- Confetti bar: 2000ms, then 500ms fade out
- Drag ghost: immediate, spring return 200ms `cubic-bezier(0.34, 1.56, 0.64, 1)` (slight overshoot)
- All respect `prefers-reduced-motion`

### Cursor and hover conventions
Keep `cursor: pointer` on all interactive elements — this is the correct Windows 11 native convention. Hover highlight on list rows uses a subtle `--bg` background change (not the accent color). Step cards use `border-color` change on hover (no background shift). This gives a native-feeling desktop app on Windows while remaining clear to the user what is clickable.

## 7. Non-Functional Requirements

- **No freeze:** All broker I/O in Rust async tasks. Svelte never blocks on `await` for network.
- **No crash:** Every `#[tauri::command]` returns `Result<T, String>`. Every Svelte component has error boundary. Every `invoke()` wrapped in `try/catch`.
- **No empty screen:** Every possible state (loading, empty, error) has a defined UI.
- **< 150ms cold start:** Maintained — no new synchronous startup operations.
- **Validation before run:** Run button disabled (grey, `not-allowed` cursor) when any validation error exists in the active flow.
